# 架构：自主运行的定制游 agent 公司（建在 macos_agent 引擎之上）

> 目标（用户设定）：把现在这家人工 B2B 定制游公司，做成一套 **agent 自主运行** 的公司，最终**完全自动化替代人工**。
> 业务权威规格：`前段获客与转化工作流程（V2）`（用户口述的六大流程，本文视其为需求真相源）。
> 定位：本目录（`company/`）是**业务层**；`../macos_agent/` 是**引擎**（在 macOS VM 里 observe→decide→act 的 ReAct 桌面代理 + broker 密钥安全 + guard VM 隔离，已真机验证：A3 计算器完整过、A1/A7/A8/M1 全绿）。**引擎不重造，业务层只做：换任务目标 + 接业务系统 + 加业务规则 + 编排流水线。**

---

## 0. 一句话架构
> **每个人工岗位 = 一个"引擎实例 + 岗位提示词 + 该岗位能操作的业务系统 + 该岗位的业务规则"。**
> 六大流程 = 一条把 lead 从"进线"推到"成交执行"的**流水线**，每一站由对应岗位 agent 处理，**碰钱/对客户/对供应商承诺的动作走人审闸门**（初期），随信心逐步放开到全自动。

---

## 1. 人工岗位 → agent 映射

| 现在的人 | agent | 操作的系统 | 主要动作 | 风险/闸门 |
|---|---|---|---|---|
| 客服（SDR） | **客服 agent** | SaleSmartly | 回消息、收 8 项需求、建信任、判意向 | 低（不碰钱）；发首价前可先人审 |
| 计调 | **计调 agent** | 价格库 + 供应商系统/网页、12306/机票站 | 按需求核算成本、生成预估报价 | 中（报价影响利润）→ 报价单人审后才发 |
| 地接匹配 | **供应商匹配 agent** | 供应商库 | 多城市地接匹配、成本×服务质量平衡 | 中 |
| 报价 | （计调 agent 的一环） | 报价单模型 | 预估报价 + 二次报价 | 中 |
| 收款 | **收款 agent** | 支付宝/PayPal 收款页 | 生成收款链接/核对到账 | 🔴高（碰钱）→ **永远人确认，不自动划钱** |
| 数据分析 | **数据 agent** | 转化数据仓 | 统计咨询/报价/成交/转化率、反馈投放 | 低 |
| 投放/内容 | **市场 agent**（最后做） | 广告平台、素材 | 内容/投放/按数据优化 | 高（花钱投广告）→ 人审预算 |

> 实现上：一个岗位 agent = `run_agent.py` 那套，但注入**该岗位的 prompts**（像 `macos/prompts.py` 那样一岗一份）+ **该岗位允许的动作/系统白名单** + **业务规则校验**。引擎的 observe/actions/guard/broker 全复用。

## 2. 系统真相源（system of record）—— 这是第一优先级
> 沿用 harness engineering 的第一教训：**"环境规格不足"才是最大瓶颈，不是模型不行。** agent 之所以能算报价、能匹配地接，前提是这些业务知识以**结构化、可查询、agent 可读**的形式存在。没有这层，再强的 agent 也只能瞎猜。所以 **Phase 0 先建数据真相源，而不是先建 agent。**

要建的结构化数据（先用本地文件/SQLite，agent 可读可写）：
1. **价格库 `price_book`**：按 `城市 × 资源类型 × 供应商 × 季节/日期段` 存成本。资源类型 = 酒店(按星级/房型/床型)、用车、导游、门票、地接、国内交通(高铁/国内机票)。
2. **供应商库 `suppliers`**：每个地接/供应商的 `报价水平 + 服务质量评分 + 稳定性 + 应急能力 + 历史客户反馈`。匹配 agent 靠它做"成本×服务"平衡。
3. **线索/需求 `leads`**：客服 agent 收集的 8 项基础信息（人数/年龄/出行日期/天数/城市/是否有机票/是否有预算/特殊要求）+ 意向分级 + 对话历史。
4. **报价单 `quotes`**：行程(多城市路线)、逐项成本、利润、**「能承诺 vs 不能承诺」标注**（星级可保证/具体酒店和加床不保证）、报价性质=预估、二次报价链路。
5. **订单/收款 `orders`**：定金比例(30%/40% 规则)、支付方式、尾款、退改政策(⚠️用户说要完善：按距入住天数分档退款——这是要补的规则表)。
6. **转化数据 `funnel`**：咨询→有效→报价→成单→金额，维度=广告内容/地区/时间/平台。

**每条数据在边界处强校验(定 what 不定 how)**——报价里的每个成本必须来自 price_book 的真实条目，不许 agent 凭空编价（这是防"假报价"的护栏，等价于 macos_agent 里"文件必须真存在"的硬判据）。

## 3. 两层安全：VM 隔离 + 业务闸门（叠加，不是二选一）
> **所有业务 agent 的运行时都跑在 macOS VM 里**（复用 macos_agent 引擎）。这正是模式四的核心用法。把两层安全分清楚，就知道为什么这样能「放心开权限」：

### 第一层｜VM 隔离（保护你的机器 + 困住注入）→ 让你敢开操作权限
- 每个岗位 agent / 每个客户会话，跑在一个 macOS VM 实例里（黄金镜像 + 跑完还原）。
- **因为炸了/被污染也困在 VM 里、还原就干净，所以敢给它开大操作权限**（自由点击/输入/跑程序）。VM 是唯一真隔离边界（spec §2A）。
- 尤其防**恶意客户消息注入**（§2A.5）：客服 agent 读的客户消息是**不可信数据**，就算被注入指挥它乱来，也出不了 VM。
- 一次性/还原让"被污染就重置"成本极低 → 可以放开手。

### 第二层｜业务闸门（保护钱 + 对客户/供应商的承诺）→ VM 拦不了这层
- ⚠️ 关键认知：VM 隔离**拦不住"业务层面的错"**。agent 用你**合法登录**的业务账号，报了错价、给客户发了错承诺、给供应商下错单——这些是「合法但错误」的动作，VM 拦不了（它保护基础设施，不保证业务正确）。
- 所以碰 **钱 / 对客户发东西 / 对供应商承诺** 的不可逆动作，另加人审闸门（初期），随"agent 判断 == 人判断"攒够记录再逐步放开。
- 💸 划钱/退款：永久人确认（最保守）。 📤 报价/消息发客户、🤝 对供应商下单、📣 投广告花预算：初期人审。

> **两层叠加**：VM 让你敢开「操作权限」，业务闸门守「业务正确性 + 钱」。互不替代。

可以先全自动的（低风险、VM 内、无对外承诺）：收集需求、内部算成本、生成报价草稿、匹配地接候选、统计转化、生成投放建议——错了还原即可。

## 4. MVP 分阶段（深度优先，先解锁地基再往上叠）
- **Phase 0｜系统真相源**：建 price_book + suppliers + leads/quotes/orders 的数据模型 + 校验。（没有它，后面 agent 全是空中楼阁。）
- **Phase 1｜客服 agent**：接 SaleSmartly，把对话 → 结构化 lead（8 项信息齐 + 意向分级）。**最高价值、最低风险(不碰钱)、最像 macos_agent 已验证的能力。** 先做它。
- **Phase 2｜计调/报价 agent**：需求 → 查 price_book/suppliers → 算成本 → 生成**预估报价单**（带「能/不能承诺」标注 + 二次报价）。报价发出前人审。
- **Phase 3｜数据 agent + 反馈闭环**：转化统计 → 投放优化建议。
- **Phase 4｜收款 agent**：生成收款链接/核对到账（划钱永远人确认）。
- **Phase 5+｜供应商下单自动化 + 市场投放 agent**：逐步把人审闸门放开。

---

## 5. 复用 vs 新建
- **直接复用**（`../macos_agent/`）：`brain/`(ReAct 循环)、`broker.py`(密钥不进 VM)、`macos/guard.py`(VM 隔离+fail-closed)、`macos/observe.py`+`actions.py`(桌面控制)。
- **业务层新建**（`company/`）：
  - `company/record/`：Phase 0 的数据真相源(price_book/suppliers/leads/quotes/orders + 校验)。
  - `company/roles/`：各岗位提示词 + 动作/系统白名单 + 业务规则(客服/计调/供应商匹配/数据…)。
  - `company/systems/`：业务系统"技能"(SaleSmartly 怎么读会话、支付页怎么观察等，基于 observe/actions 封装)。
  - `company/pipeline.py`：把 lead 从进线推到执行的编排 + 人审闸门。
- **治理层**：沿用刚写的 `macos_agent/docs/harness-hardening-spec.md` 那套(AGENTS.md/不变量 lint/GC)，扩到 company/，让业务层也能被 agent 可靠扩展。

## 6. 未决/要用户拍板（不阻塞 Phase 0）
- SaleSmartly：有没有 API？还是只能网页操作(那就走 macos_agent 桌面控制)？决定客服 agent 怎么接。
- 退改政策规则表：用户说要完善"按距入住天数分档退款"——这是 orders 模型里要填的真实规则，需要你给数字。
- 价格库来源：现在价格/供应商数据在哪(Excel？群里？人脑？)——决定 Phase 0 怎么把它结构化。
- 低价引流→二次报价：首价故意压低、按客人感觉再报——这套"报价策略"要写成规则，否则 agent 不知道怎么定第一口价。

## 7. 下一步
Phase 0（系统真相源）是给 Codex 的第一张工单：它是**纯数据建模 + 校验代码**——Codex 在普通开发环境敲代码就行，**这一步"写代码"本身不需要在 VM 里**（跟写 lint 工具一样）。注意区分：**业务 agent 跑起来时全部在 VM 里**（§3 第一层），只是"开发这些代码"这个动作不需要 VM。写完工单 → 发 Codex → Claude 审（同 macos_agent 的循环）。
